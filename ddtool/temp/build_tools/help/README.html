Introduction : 
 
            DD tool will parse the data dictionary tool and generate calss file containing the definition of each table. 

Scope :

         Scope of this document is intended for integration team/developers to understand the usage of the tool.

Purpose of the tool :
 
   
         In Web NMS 5 we  have moved to Data Object  Model. Here every column needs to accessed as string. To provide compile time check an constant file needs to be generated.

For example consider below node of data-dictionary

  <table name="ModelType">
        <description>GVF MT</description>
                <columns>
                        <column name="NAME">
                               ....
                        </column>
                        <column name="DISPLAYNAME">
                                ...
                        </column>
                        <column name="DATATYPE">
                                 ...   
                        </column>
                             ....
                    </columns>
  <primary-key name="ModelType_PK">
                            <primary-key-column>NAME</primary-key-column>
                </primary-key>  

               <foreign-keys>
     ....
        </table>
   An tool that parses the data-dictionary and an java file has to be generated for each and every table. i.e. above example node has to be parsed and generated as java file

package com.adventnet.nms.gvf;
public class final interface ModelType
{
       private ModelType() { }
       public static final String NAME= "NAME" ;
       public static final String SUBTYPE= "SUBTYPE" ;
       public static final String DISPLAYNAME= "DISPLAYNAME" ;
       public static final String DATATYPE= "DATATYPE" ;
       public static final String HANDLERCLASS= "HANDLERCLASS" ;
       public static final String CRID= "CRID" ;
       public static fianl String TABLE="ModelType";
}


  So application developers can generate constant file at development time and refer the constants in their application (e.g. "ModelType.NAME" ) , when they are operating on data Object API. Compile time check of the column names will be accomplished by generated constant file.
   
  Since the Generated Constant files are generated from data-dictionary.xml. It need not be integrated in CVS source repository. In nms5 build this constant file has to be generated from all data dictionary file , generated files has to be compiled and to be bundled along with nms5 binary as an jar.
  Then application code has to be compiled with generated constant file jar in classpath. Additionally javadoc file needs to be generated from "constant File". Because tool will generate file with proper java doc comments according to data dictionary definition.

Usage :
    
Set classpath as described below,
  •  classpath settings :
    • Velocity Jars
      • lib/velocity-1.3.1.jar
      • lib/velocity-dep-1.3.1.jar
    • Command  Line Parser Jars
      • lib/commons-cli-1.0.jar
    • Tools Jar
      • lib/AdventNetDDTools.jar
  • Running the tool
    • To know about the different parameters to be used run the tools with help argument as follows ,
      • java  test.DDReader  -help.
        • Above will result in below out put
        • usage: DD Tool
           -dd <file>            DataDictionary File to be parsed. <Mandatory>
           -help                 print this message
           -outDir <directory>   Root directory in which out files needs to be created. <Mandatory>
           -template <file>      File location of the Template to be used for file generation. <Mandatory>
      • As in the above help statements provide inputs, For example consider below example ,
        • java test.DDReader -dd c:\Development\3g\k\ms\performance\gvf\fe\conf\data-dictionary.xml  -template c:\Development\3g\k\tools\ddtool\resources\conf\ClassTemplate.vtl -outDir output
  •   OutPut
    • As per above input  set of java files will be generated.
      • with classname of Tablename in uppercase.
      • Package name will be generated as follows , com.adventnet.<modulename specified in data-dictionary.xml>
      • Files will be created under directory of outDir + pkgNam(as per example it is of  output\com\adventnet\gvf where gvf is module name).
    • Generated Java file can be compiled and referred in the application as constants , for example  "MODELTYPE.NAME"
    • Even java doc can be generated for the generated java file. It will have detailed definition as per the data dictionary definition.
